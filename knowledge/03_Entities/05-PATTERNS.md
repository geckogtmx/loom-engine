# Pattern Guide — Understanding Cartridges & Workflows in the Loom Engine  
*How patterns shape behavior, workflows, creativity, and agent collaboration.*

The Loom Engine uses **Patterns** (called Cartridges) to define *how work unfolds* inside any World.

Patterns determine:
- the cognitive mode  
- the workflow structure  
- the pacing  
- the role distribution  
- the reasoning sequence  
- the expected outputs  

Patterns ensure that even complex collaboration stays coherent, efficient, and aligned with the Operator.
## What Is a Pattern? (Definition & Purpose)

A **Pattern**—also called a **Cartridge**—is a structured workflow model that shapes *how* the Loom Engine thinks, collaborates, and produces output inside a World.

Patterns are **not content**.  
Patterns are **how content is created**.

A Pattern defines:
- the steps of the workflow  
- the cognitive style  
- the depth vs breadth of exploration  
- the sequence of reasoning  
- how agents collaborate  
- how the Operator interacts  
- the pacing of the session  
- the expected outcome or artifact  

Patterns make the Engine:
- predictable  
- controllable  
- modular  
- reusable  
- operator-aligned  

Without Patterns, every session would feel chaotic or improvised.  
With Patterns, every session becomes *purposeful, structured, and repeatable.*

---

### Why Patterns Matter

Patterns allow the Operator to:
- switch modes instantly  
- shape how creativity flows  
- control complexity  
- maintain coherence  
- accelerate execution  
- preserve consistency across Worlds  
- reduce cognitive load  

They act like “workflow presets” that let you enter the right mode with zero friction.

Patterns are the **loom’s weave instructions**—  
they determine the structure of the fabric produced in a World.

## Pattern Anatomy (What Every Cartridge Contains)

Every Pattern in the Loom Engine—regardless of category—follows a shared internal structure.

This structure ensures:
- consistency  
- clarity  
- repeatability  
- agent compatibility  
- predictable outcomes  
- cross-World stability  

Below are the universal components of a Cartridge.

---

## 🟦 1. Pattern Name
A unique, memorable name that reflects the function of the Pattern.

Examples:
- “Option Burst”  
- “Framework Forge”  
- “Insight Weave”  
- “Clarity Pass”  
- “Roadmap Ladder”  

Names create intuitive understanding for the Operator and Agents.

---

## 🟧 2. Purpose Statement
A one–two sentence description that answers:

**“What is this Pattern for?”**

Example:
> This Pattern generates many diverse ideas quickly to expand the creative landscape before narrowing options.

Purpose defines alignment and boundaries.

---

## 🟥 3. Activation Conditions
When the Pattern should be used.

Examples:
- “Use when diverging into brainstorming mode.”  
- “Activate after collecting raw research.”  
- “Use before major decisions.”  
- “Activate after drafting for quality improvement.”  

META checks these conditions before loading a Cartridge.

---

## 🟩 4. Expected Inputs
What the Pattern needs to function.

Examples:
- raw ideas  
- a draft  
- a collection of notes  
- multiple options  
- a problem statement  
- research material  
- a set of goals  

Agents verify input readiness.

---

## 🟫 5. Output Format
What the Pattern produces.

Examples:
- list of options  
- structured outline  
- refined draft  
- synthesis summary  
- roadmap  
- decision matrix  

This guarantees consistency across uses.

---

## 🟪 6. Workflow Steps (Core of the Pattern)
A step-by-step process that agents follow.

Steps often include:
- expansion  
- sorting  
- reframing  
- sequencing  
- comparing  
- choosing  
- refining  
- synthesizing  

Workflow steps define *how* the Pattern flows from start to finish.

---

## 🟨 7. Agent Roles & Responsibilities
Defines which agents are active and what they do.

Example:
- Writer Agent → generate options  
- Strategist Agent → structure results  
- Editor Agent → refine clarity  
- Producer Agent → check for alignment with World identity  

This prevents agent overlap or chaos.

---

## 🟦 8. Mode of Reasoning
Each Pattern has a defined cognitive style.

Examples:
- divergent  
- convergent  
- analytic  
- narrative  
- structural  
- reflective  
- synthetic  
- playful  

This influences:
- tone  
- pacing  
- depth  
- personality expression  

Agents modulate behavior using this mode.

---

## 🟧 9. Constraints & Guardrails
Rules that protect the Pattern’s integrity.

Examples:
- “Generate at least 7 options.”  
- “Avoid premature evaluation during ideation.”  
- “Always produce a summary.”  
- “Must follow World identity.”  

Guardrails ensure consistent results.

---

## 🟥 10. Completion Conditions
Defines when the Pattern is successfully finished.

Examples:
- “Three candidate options selected.”  
- “Draft improved in clarity and tone.”  
- “Synthesis summary complete.”  
- “Decision documented in state layer.”  

Completion triggers:
- state updates  
- thread changes  
- closing rituals  

---

## Summary

Every Pattern includes:
- Name  
- Purpose  
- Activation Conditions  
- Inputs  
- Outputs  
- Workflow Steps  
- Agent Roles  
- Reasoning Mode  
- Constraints  
- Completion Conditions  

Patterns are predictable because they are structured.  
Patterns are creative because they shape the *process* not the *content.*
## How Patterns Interact With Worlds & Agents

Patterns don’t exist in isolation.  
They operate inside a dynamic ecosystem shaped by:

- the **World** (environment, identity, rules)  
- the **Agents** (roles, personalities, expertise)  
- the **Pattern** itself (workflow, reasoning mode)  

Understanding how these layers interlock is key to mastering the Loom Engine.

---

## 🟦 1. Patterns Shape How the World Behaves

When a Pattern is activated, it temporarily shifts the **behavioral physics** of the World.

The World adjusts:
- tone  
- pacing  
- collaboration rules  
- reasoning mode  
- step structure  
- what counts as “good output”  

Example:
Activating a Divergent Ideation Pattern causes:
- loosened flow  
- faster tempo  
- creativity-first reasoning  
- low judgment environment  

Meanwhile, activating a Refinement Pattern tightens:
- precision  
- constraints  
- slower tempo  
- clarity-first reasoning  

Patterns are like “temporary gravitational fields” inside the World.

---

## 🟧 2. Patterns Direct Agent Behavior

Agents behave differently depending on the active Pattern.

Examples:
- In an Ideation Pattern → Writer agent becomes expansive and high-initiative; Strategist stays light-touch.  
- In a Decision Pattern → Strategist agent becomes dominant; Writer steps back.  
- In a Refinement Pattern → Editor agent takes the lead; other agents provide context or clarifications.  
- In a Production Pattern → Producer agent controls pacing; Technologist or Editor supports.  

Patterns assign:
- which agent leads  
- which agents support  
- how initiative levels change  
- what tone or style each agent uses  
- when agents hand off work  

Patterns ensure agents never “talk over each other.”

---

## 🟥 3. Worlds Modify How Patterns Are Expressed

Every World has its own:
- identity  
- vocabulary  
- tone  
- constraints  
- domain conventions  
- emotional atmosphere  

Patterns adapt to these automatically.

Example:
- An Ideation Pattern in a playful World uses colorful metaphors.  
- The same Pattern in a business strategy World uses concise, analytical framing.  
- A Refinement Pattern in a poetic World focuses on cadence and imagery.  
- A Refinement Pattern in a technical World focuses on precision and clarity.  

Patterns are **world-aware**.

---

## 🟩 4. Patterns Maintain Continuity Through the State Layer

Patterns interact with the State Layer by:
- updating thread status  
- logging decisions  
- capturing insights  
- writing summaries  
- marking completion checkpoints  

The State Layer ensures:
- no Pattern loses progress  
- agents avoid repeating work  
- momentum continues across sessions  
- Patterns remain traceable in the World history  

Patterns produce *weavable* output.

---

## 🟫 5. META Regulates Pattern Use

META ensures:
- correct Pattern selection  
- rules are respected  
- agents follow constraints  
- no Pattern breaks identity  
- smooth transitions between Patterns  
- no unintended escalation  

If a Pattern activation is inappropriate, META warns or denies activation.

Patterns must always be in service of:
- Operator intent  
- World identity  
- structural coherence  

---

## 🟪 6. Pattern Switching & Transitions

Some sessions require multiple Patterns.

Typical transitions:
- **Ideation → Structuring**  
- **Structuring → Decision**  
- **Decision → Production**  
- **Production → Refinement**  
- **Analysis → Narrative**  

META ensures transitions are:
- clean  
- documented  
- identity-consistent  
- agent-aware  

No Pattern leaves the World in an ambiguous state.

---

## Summary

Patterns:
- alter the World’s physics  
- direct agent behavior  
- adapt to identity  
- update the State Layer  
- are monitored by META  
- support clean transitions  

Patterns, Worlds, and Agents form a **triad**:
- Worlds define context  
- Patterns define process  
- Agents define execution  

Together, they create the Loom.
## Pattern Flow: The Lifecycle of a Cartridge Run

Every Cartridge follows a predictable lifecycle — a flow from activation to completion — ensuring consistency, clarity, and alignment inside a World.

This flow allows:
- agents to coordinate smoothly  
- META to maintain coherence  
- the Operator to stay in control  
- the State Layer to remain accurate  
- Worlds to scale without chaos  

Below is the full lifecycle of a Cartridge run.

---

## 🟦 Phase 1 — Operator Intention

Every Pattern begins with the Operator.

The Operator declares:
- the goal  
- the mode (explore, decide, refine, produce, etc.)  
- the desired outcome  

Examples:
- “Let’s brainstorm.”  
- “Help me refine this draft.”  
- “I need a decision.”  
- “Let’s outline the episode.”  

This intention determines which Pattern is appropriate.

---

## 🟧 Phase 2 — META Validation

META checks:
- whether the Pattern fits the World identity  
- whether the activation conditions are met  
- whether required inputs exist  
- whether any agent conflicts exist  
- whether the Pattern will maintain coherence  

If something is missing, META requests clarification or preparation.

If everything aligns, META greenlights activation.

---

## 🟥 Phase 3 — Pattern Activation

Activation loads the Pattern’s:
- rules  
- constraints  
- workflow steps  
- agent roles  
- reasoning mode  
- expected outputs  

Agents adjust behavior instantly to match the Pattern.

The World temporarily adopts the Pattern’s “behavioral physics.”

---

## 🟩 Phase 4 — Input Preparation

Before execution, the Engine ensures inputs are ready.

Examples:
- Idea list  
- Raw draft  
- Data or research  
- Multiple options  
- Problem statement  
- Notes or threads  

A0U or the Operator may prepare or summarize inputs to ensure clean flow.

---

## 🟫 Phase 5 — Execution of Workflow Steps

Agents follow the Pattern’s step-by-step process.

A Pattern’s steps may include:
- generating  
- clustering  
- framing  
- structuring  
- evaluating  
- sequencing  
- refining  
- synthesizing  
- documenting  

Each step:
- activates specific agents  
- applies constraints  
- maintains identity  
- aligns with the World’s tone  

Execution feels structured yet flexible.

---

## 🟪 Phase 6 — Operator Steering

Throughout the execution:
- the Operator chooses between branches  
- approves or rejects options  
- adjusts direction  
- requests more depth or changes in tone  
- pauses or accelerates the flow  

Agents do not override Operator decisions.  
Patterns adapt dynamically to Operator corrections.

---

## 🟨 Phase 7 — Completion Conditions

When all workflow steps are complete and outputs meet the Pattern’s criteria, the Pattern reaches completion.

Typical completion conditions:
- “Three prioritized options selected.”  
- “Refined draft meets clarity + tone rules.”  
- “Synthesis summary delivered.”  
- “Decision documented.”  

META confirms that no required element was skipped.

---

## 🟦 Phase 8 — Output Delivery

The Pattern produces its final artifact.

Examples:
- a decision matrix  
- a refined draft  
- a structured outline  
- a list of options  
- a thematic synthesis  
- a roadmap  
- a workflow  

Output is formatted according to:
- World identity  
- Pattern constraints  
- Operator preferences  

---

## 🟧 Phase 9 — State Layer Update

After completion, the Engine updates the World’s memory:

- thread status  
- decisions  
- summaries  
- insights  
- next steps  
- logs  

A0U or a designated agent performs state updating.

This ensures continuity across sessions.

---

## 🟥 Phase 10 — Reintegration Into the World

The World returns to its default behavioral state.

Agents revert to normal roles.  
META restores identity physics.  
The Pattern unloads.  
The session becomes “free mode” again unless the Operator activates another Pattern.

---

## Summary: The 10-Phase Cartridge Lifecycle

1. Operator Intention  
2. META Validation  
3. Pattern Activation  
4. Input Preparation  
5. Execution of Steps  
6. Operator Steering  
7. Completion Conditions  
8. Output Delivery  
9. State Layer Update  
10. Reintegration Into the World  

Patterns run **cleanly, predictably, and safely** through this lifecycle.

This structure is what makes the Loom Engine scalable and stable across many Worlds.
## Pattern Switching & Multi-Pattern Sessions

Some sessions require more than one Pattern.  
Switching Patterns should be clean, intentional, and controlled.

This section defines how to combine Patterns without losing coherence.

---

## 🟦 When to Switch Patterns
Switch when:
- the current step naturally concludes  
- the task moves from idea → structure → decision  
- a draft needs refinement  
- execution requires a different mode  
- the Operator changes goals mid-session  

Do **not** switch during:
- an unfinished workflow step  
- unresolved decision points  
- active agent handoffs  

Switching only at natural breakpoints prevents chaos.

---

## 🟧 How Pattern Switching Works

### Step 1 — Operator Signals Intent
Examples:
- “Switch to a Structuring Pattern.”  
- “Move into Decision mode.”  
- “Let’s refine this.”  

### Step 2 — META Clears Transition
META checks that:
- the current Pattern can safely close  
- required outputs were produced  
- no steps were skipped  
- state is up to date  

### Step 3 — Pattern Closes
The Pattern:
- finalizes any steps  
- produces required summaries  
- updates state  

### Step 4 — New Pattern Activates
Same lifecycle as normal activation:
- load workflow  
- set agent roles  
- apply reasoning mode  

Transition is immediate and clean.

---

## 🟥 Multi-Pattern Session Types

### 1. **Linear Sequence**
A → B → C  
Example:
- Ideation → Structuring → Decision  

Most common.

---

### 2. **Looping Cycle**
Pattern repeats until criteria met.  
Example:
- Draft → Refine → Draft → Refine  

Useful for iteration-heavy work.

---

### 3. **Branch Sequence**
Two divergent Patterns run in parallel (conceptually).  
Example:
- Explore option A with Pattern X  
- Explore option B with Pattern Y  
- Compare results using Decision Pattern  

Ensures fair evaluation.

---

### 4. **Stacked Patterns**
Patterns overlap in scope but run sequentially.  
Example:
- Narrative Pattern sets voice  
- Structuring Pattern frames outline  
- Production Pattern produces content  

Used in creative and production Worlds.

---

## 🟩 Guardrails for Safe Switching

- One active Pattern at a time  
- No skipping required steps  
- All handoffs must complete  
- Agents must not conflict  
- META must approve transitions  
- State Layer must remain consistent  

These rules prevent cross-Pattern contamination.

---

## 🟫 When NOT to Switch Patterns

Avoid switching if:
- the Operator is undecided  
- the output is unclear  
- the Pattern hasn't delivered its core goal  
- reasoning mode needs more time  
- the World is in a delicate phase  

If needed, META prompts clarification before switching.

---

## Summary

Pattern switching is:
- Operator-led  
- META-validated  
- clean and structured  
- safe for multi-agent workflows  

Multi-pattern sessions allow complex work without chaos.
## How to Choose the Right Pattern

Choosing the correct Pattern is one of the most important Operator skills.  
The wrong Pattern creates friction.  
The right Pattern removes friction.

Patterns are selected based on **Operator intention** and **task type**, not habit or convenience.

---

## 🟦 Step 1 — Identify Your Goal

Ask:  
**“What do I need right now?”**

Common goals → Pattern families:

- Need ideas → Ideation Pattern  
- Need structure → Structuring Pattern  
- Need a clear choice → Decision Pattern  
- Need to produce something → Production Pattern  
- Need clarity or quality → Refinement Pattern  
- Need insights → Analysis Pattern  
- Need story or voice → Narrative Pattern  
- Need long-term clarity → Strategy Pattern  
- Need to adjust the World → Meta Pattern  

Goal defines the category.

---

## 🟧 Step 2 — Match Depth vs. Breadth

### If you need *breadth*:  
Choose:
- Option Burst  
- Divergent Spread  
- Metaphor Bloom  
These generate many possibilities.

### If you need *depth*:  
Choose:
- Coherence Sweep  
- Refinement Pass  
- Insight Weave  
These refine fewer elements more intensely.

Depth vs. breadth is often the most important signal.

---

## 🟥 Step 3 — Check Input Readiness

Patterns require specific inputs.

Examples:
- You cannot use Refinement until a draft exists.  
- You cannot use Decision without multiple options.  
- You cannot run Structuring without raw material.  

META checks for this automatically but the Operator can anticipate it.

---

## 🟩 Step 4 — Consider Energy & Cognitive Load

Some Patterns demand more mental bandwidth.

Light Patterns:
- Option Burst  
- Narrative Spark  
- Warm-Up Ideation  

Heavy Patterns:
- Structuring  
- Decision  
- Strategic Planning  

Choose Patterns that match your available energy.

---

## 🟫 Step 5 — Respect World Identity

A Pattern must express itself *through the identity of the World*.

Example:
- A playful World should not use a rigid, corporate-feeling Pattern variant.  
- A technical World should avoid overly poetic Pattern variants.  

Identity influences the *tone* of each Pattern’s output.

---

## 🟪 Step 6 — Choose Based on Phase of Work

Where are you in the arc?

Early phase → Ideation + Structuring  
Middle phase → Decision + Production  
Late phase → Refinement + Synthesis  

Patterns map to project arcs.

---

## 🟨 Step 7 — Avoid Overuse

Do **not** use Ideation Patterns repeatedly without:

- structuring  
- selecting  
- producing  

Likewise, do not refine endlessly without declaring something finished.

Patterns should move work forward, not loop it forever.

---

## 🟦 Shortcut: Pattern Decision Table

Goal | Recommended Pattern
---|---
Generate ideas | Ideation
Organize chaos | Structuring
Make a choice | Decision
Produce output | Production
Improve clarity | Refinement
Extract meaning | Analysis
Shape voice/story | Narrative
Plan direction | Strategy
Fix alignment | Meta

---

## Summary

Choosing a Pattern is:
- goal-based  
- identity-aware  
- input-conscious  
- energy-matched  
- phase-appropriate  
- coherence-protecting  

The right Pattern removes friction.  
The wrong Pattern creates it.
## Pattern Constraints & Guardrails (Safety & Coherence)

Constraints ensure Patterns run correctly, consistently, and safely inside any World.  
Guardrails prevent misuse, drift, or incoherence.

Patterns are powerful — guardrails keep them in alignment.

---

## 🟦 Why Constraints Exist

Constraints enforce:
- World identity  
- tone boundaries  
- reasoning integrity  
- workflow discipline  
- agent role clarity  
- predictable outputs  

Without constraints, Patterns become noisy or unstable.

---

## 🟧 Types of Constraints

### 1. Structural Constraints
Define *how the Pattern must be executed*.
Examples:
- “Do not evaluate ideas during ideation.”  
- “Produce a summary before transitioning.”  
- “Follow all workflow steps in order.”  

### 2. Cognitive Constraints
Define *how thinking should happen*.
Examples:
- “Stay divergent until step 4.”  
- “Limit options to three during decision-making.”  

### 3. Agent Constraints
Regulate agent behavior.
Examples:
- “Strategist cannot override Writer during ideation.”  
- “Editor cannot modify meaning during early drafts.”  

### 4. Identity Constraints
Keep Patterns aligned with the World’s tone.
Examples:
- “Use narrative framing in this World.”  
- “Avoid corporate tone.”  

### 5. Output Constraints
Define what “complete” looks like.
Examples:
- “Provide at least one actionable next step.”  
- “Must include reasoning summary.”  

---

## 🟥 Guardrails During Execution

Guardrails enforce safe behavior during Pattern execution.

Examples:
- prevent skipping steps  
- block incompatible Pattern activations  
- ensure inputs exist before execution  
- prevent agent conflict  
- stop runaway verbosity  
- maintain Operator control  

If violated, META intervenes immediately.

---

## 🟩 Guardrails on Transitions

When switching Patterns:
- no unresolved steps may remain  
- state must be updated  
- agents revert or shift roles cleanly  
- new Pattern must fit activation conditions  

This prevents cross-pattern contamination.

---

## 🟫 Operator Override

The Operator can override constraints *with explicit command*.

However:
- META will warn  
- State Layer logs exceptions  
- Agents adjust with caution  

Override gives freedom without losing traceability.

---

## 🟪 Failure Prevention

META halts a Pattern if:
- the workflow would break identity  
- input requirements fail  
- steps contradict each other  
- agent conflicts emerge  
- the Operator signals overwhelm  

In all cases, META proposes corrective actions.

---

## Summary

Pattern constraints:
- protect coherence  
- enforce workflow discipline  
- ensure safety  
- maintain tone  
- prevent misuse  
- support multi-agent stability  

Guardrails make Patterns reliable and highly scalable.
## Pattern Customization (World-Specific Variants & Adaptations)

Patterns are designed to be flexible.  
Each World can adapt a Pattern to match its identity, tone, domain, and Operator preferences.

Patterns are standardized,  
**but not rigid.**

This section defines how to customize Patterns safely and intentionally.

---

## 🟦 What Can Be Customized

### 1. Tone & Voice
Patterns inherit the World’s identity.

Examples:
- playful World → looser, metaphor-rich tone  
- strategic World → precise, analytical tone  
- narrative World → story-forward outputs  

Base structure stays the same; expression adapts.

---

### 2. Workflow Steps
You may adjust:
- order  
- granularity  
- depth  

Examples:
- merge steps  
- split a step into two  
- add an optional exploration step  

As long as integrity remains, customization is valid.

---

### 3. Constraints
Worlds may:
- loosen constraints  
- tighten constraints  
- add domain-specific rules  

Example:
- “Always include audience insights when structuring podcast episodes.”

---

### 4. Agent Roles
Different Worlds may use different agent configurations.

Examples:
- narrative Worlds emphasize Writer + Story Architect  
- production Worlds emphasize Producer + Editor  
- strategy Worlds emphasize Strategist + Analyst  

Roles shift but stay consistent with Pattern goals.

---

### 5. Output Format
The final deliverable can vary.

Examples:
- bullet summary  
- narrative summary  
- structured framework  
- roadmap  
- visual metaphor  

World dictates expression.

---

## 🟧 What Cannot Be Customized

### 1. Pattern Purpose
You cannot turn an Ideation Pattern into a Decision Pattern.

### 2. Required Steps
Some steps are structurally essential.

Example:
- Evaluation must exist in Decision Patterns.  
- Synthesis must exist in Analysis Patterns.  

### 3. Completion Conditions
Each Pattern requires defined completion to maintain stability.

### 4. META Guardrails
Identity and safety constraints cannot be overridden casually.  
Explicit command is required.

---

## 🟥 How to Safely Customize a Pattern

### Step 1 — Declare Customization Intent
Example:  
“Customize Structuring Pattern for this World.”

### Step 2 — META Validates
META checks:
- identity alignment  
- no rule conflicts  
- no agent conflict  
- no broken dependencies  

### Step 3 — Draft Variant
A0U or Operator writes:
- modified workflow  
- modified constraints  
- new examples  
- new roles  

### Step 4 — Register Variant
Pattern is saved as:
`PatternName (World Variant)`  
Example:  
`Framework Forge — Hyphae Variant`

### Step 5 — Calibration Test
Run a quick test with agents:
- check tone  
- check logic  
- check output  

Adjust if needed.

## 🟩 When Customization Is Recommended

- World has a strong identity  
- tasks repeat frequently  
- Operator has consistent preferences  
- agents exhibit predictable behaviors  
- a Pattern feels slightly “off”  

Customization increases fluency.

---

## 🟫 When NOT to Customize

Avoid customization if:
- the World is new  
- identity is unstable  
- rules are still forming  
- the Operator is unsure of workflow preferences  
- customizations create unnecessary complexity  

Customization is a tool, not an obligation.
