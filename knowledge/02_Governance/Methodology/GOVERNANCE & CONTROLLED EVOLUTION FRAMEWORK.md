## 1. META — Governance Root of LOOM

### 1.1 Definition

**META** is the **root governance layer** of **LOOM**.

META is:

- a constitutional rule system
    
- a governance authority
    
- a validation and authorization layer
    

META is **not**:

- an Agent
    
- an actor
    
- a learner
    
- a creative entity
    
- an executor
    

META **never produces content** and **never performs tasks**.

META defines **what may exist, what may change, and what may persist** within LOOM.

---

### 1.2 META Responsibilities

META is solely responsible for:

- defining global constraints
    
- preserving identity coherence (L4 integrity)
    
- detecting structural or behavioral drift
    
- authorizing or rejecting evolution
    
- validating system-level integrity
    
- halting unsafe or misaligned processes
    

META **does not**:

- create Agents
    
- tune Agents
    
- deploy Agents
    
- execute workflows
    
- participate in Worlds
    

META has **authority without execution**.

---

### 1.3 META as Constitutional Layer

META functions as the **constitutional layer** of LOOM:

- it defines the rules
    
- it interprets violations
    
- it authorizes exceptions
    
- it preserves long-term coherence
    

All systems below META operate **only within explicit authorization**.

No system may bypass META.

---

## 2. Authorized Subsystems (Under META)

META authorizes **three subordinate systems**.

Each system has **non-overlapping authority**.

No system may assume another’s role.

---

## 3. A0 (Agent Zero) — Enforcement & Execution Layer

### 3.1 Definition

**A0 (Agent Zero)** is a **special-purpose Agent** authorized by META to:

- enforce governance rules
    
- execute approved structural actions
    
- supervise other Agents at runtime
    

A0 is **not a peer Agent**.

---

### 3.2 A0 Responsibilities

A0 is responsible for:

- deploying META-approved Agent versions
    
- enforcing version isolation
    
- supervising live Agent behavior
    
- escalating violations to META
    
- coordinating Agent interactions
    
- applying certified changes
    

A0 **does not**:

- define policy
    
- approve evolution
    
- modify Agent specifications
    
- self-evolve
    

A0 enforces.  
A0 does not decide.

---

### 3.3 A0 Authority Limits

A0:

- acts **only** under META authorization
    
- cannot expand its own scope
    
- cannot redefine rules
    
- cannot override Telos
    

If META is the law,  
**A0 is the enforcement mechanism**.

---

## 4. Agent University — Specification & Certification Layer

### 4.1 Definition

The **Agent University** is a **formal specification and certification system**.

It is **not an Agent**.

---

### 4.2 University Responsibilities

The University is responsible for:

- defining Agent specification schemas
    
- designing Agent versions
    
- certifying Agent configurations
    
- maintaining lineage records
    
- regulating allowed evolution deltas
    

The University **does not**:

- deploy Agents
    
- supervise live behavior
    
- enforce rules
    
- approve evolution scope
    

---

### 4.3 Agent Versions

Agent versions are:

- immutable once issued
    
- non-hierarchical
    
- non-evaluative
    

A new version is **different**, not “better”.

Agents **never mutate in place**.

---

## 5. Controlled Agent Evolution (META-Led)

### 5.1 Evolution Authority

Only **META** may authorize evolution.

Evolution may be triggered by:

- explicit Operator intent
    
- drift detection
    
- architectural refactoring
    
- validation outcomes
    

Agents **cannot** initiate evolution.

---

### 5.2 Evolution Flow (Canonical)

`META authorizes scope         ↓ Agent University defines specification         ↓ University certifies version         ↓ META validates compliance         ↓ A0 deploys approved version`

All prior versions remain archived and addressable.

## 6. Replication Layer — Validation & Proof System

### 6.1 Definition

The **Replication Layer** is a **validation system**, not a generative system.

It exists to:

- test architectural assumptions
    
- validate governance constraints
    
- compare structured vs unstructured cognition
    
- surface unintended behavior
    

---

### 6.2 Replication Authority

Replication:

- observes
    
- measures
    
- compares
    
- reports
    

Replication **does not**:

- change the system
    
- evolve Agents
    
- deploy results
    
- override META
    

---

### 6.3 Replication as META Instrument

Replication exists **for META**.

META may use Replication outputs to:

- authorize change
    
- deny evolution
    
- refine constraints
    
- validate architecture
    

Replication never bypasses governance.
