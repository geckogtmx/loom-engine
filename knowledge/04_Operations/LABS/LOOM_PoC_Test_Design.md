# LOOM Engine: Contextual Memory Injection Proof-of-Concept (PoC)

## 1. Executive Summary
This document outlines the design and methodology for a Proof-of-Concept (PoC) aimed at validating the core hypothesis of the LOOM Engine: that **structured, persistent memory layers (L1-L4) significantly improve LLM output quality, relevance, and alignment compared to standard "natural" prompting.**

We are building a **LOOM Prompt Lab**, a controlled testing environment to isolate and quantify the value added by the LOOM Cognitive Architecture, specifically the memory injection mechanism.

## 2. Rationale: Why this Test?
The LOOM Engine relies on a complex architecture of "Agents" and "Memory Layers." Before building the full automated infrastructure, we must validate the utility of its fundamental component—the Memory Stack.

This PoC serves to:
*   **Validate the Hypothesis:** Does injecting L1-L4 context actually produce better results than a skilled user prompting naturally?
*   **Quantify "Better":** We need to move beyond "it feels better" to concrete observations (e.g., "The agent remembered the coding style," "The agent avoided a previously known error").
*   **Model Agnosticism:** By testing across multiple models (GPT, Gemini, Claude), we prove that LOOM is a universal framework, not dependent on a specific model's "native" memory.
*   **Define "Context Economy":** Help us understand how much token overhead is required to achieve a noticeable improvement in reasoning.

## 3. Test Methodology

### 3.1 The A/B Test Structure
Every test run consists of a single **Task** executed simultaneously under two conditions:
1.  **Condition A (Control - Natural):** The model receives only the user's specific prompt.
2.  **Condition B (Experimental - Assisted):** The model receives the same user prompt, PREPENDED with a structured injection of L1-L4 memory context derived from a specific "Agent Profile."

### 3.2 Data Volume & Statistical Significance
For a valid PoC, we do not need Big Data; we need *Deep Data*.
*   **Recommended Volume:**
    *   **3 Distinct Agent Profiles:** (e.g., "The Coder," "The Philosopher," "The Project Manager"). This ensures the memory works for different domains.
    *   **10-15 Unique Prompts per Profile:** Covering various tasks (Retrieval, Creative Generation, Logic/Debugging).
    *   **Total Data Points:** ~30-45 paired comparisons.
*   **Justification:** This volume is small enough to allow for detailed **human qualitative review** (reading every word) but large enough to filter out "lucky guesses" by the model. If the Memory-Assisted prompt wins 30 out of 45 times, we have a strong signal.

## 4. Technical Architecture: The "LOOM Prompt Lab"

We will develop a lightweight, local application to serve as the test bench.

*   **Stack:** Python + Streamlit.
*   **Interface:** A web-based dashboard running locally on your machine.
*   **Data Source:** JSON files stored in a `/mocks` folder. This allows for rapid editing of "Memory" without touching code.
*   **Logging:** A CSV file (`experiment_logs.csv`) acts as the database, recording inputs, outputs, timestamps, and model settings.

### User Workflow
1.  **Setup:** User edits a `profile.json` to define the Agent's current state (L1-L4).
2.  **Launch:** User starts the Lab App via CLI (`streamlit run app.py`).
3.  **Config:** User selects the Profile and the AI Model to test.
4.  **Execute:** User types a prompt (e.g., "Write a python script to parse this log") and hits "RUN".
5.  **Review:** The app displays both outputs side-by-side.
6.  **Analysis:** User opens the generated CSV in Excel to grade and annotate the results.
