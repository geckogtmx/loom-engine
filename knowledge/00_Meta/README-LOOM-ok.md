# LOOM Engine

**LOOM** is an **Operator-first cognitive architecture** for working with AI across long time horizons without losing context, identity, or control.

It is designed for people who already understand LLMs, model/context separation, and the kinds of problems MCP-style systems are trying to solve.

LOOM is **not** a chatbot framework, not an autonomous agent system, and not a project manager.  
It is a **governed interaction framework** for thinking and building with AI over time.

---

## What LOOM Is (and Is Not)

**LOOM is:**

- a structured execution environment
    
- centered on a human Operator
    
- explicitly governed
    
- context-first, not prompt-first
    
- designed for continuity across sessions, weeks, and months
    

**LOOM is not:**

- an autonomous system
    
- a self-learning agent loop
    
- a tool orchestration playground
    
- a memory blob that “just grows”
    
- a replacement for human judgment
    

LOOM assumes that **clarity requires structure**, and that long-term AI collaboration fails without explicit boundaries.

---

## The Problem LOOM Addresses

Most AI interactions break down over time due to:

- uncontrolled context growth
    
- implicit memory
    
- agent drift
    
- unclear authority
    
- identity dilution
    
- short-term optimization
    

These failures are not model limitations — they are **architectural failures**.

LOOM exists to provide:

- explicit context isolation
    
- governed execution
    
- stable identity
    
- auditable continuity
    
- predictable AI behavior across sessions
    

---

## The LOOM Mental Model

LOOM organizes work using a small number of strict primitives.

### Worlds

A **World** is an isolated project environment.

Each World has:

- a defined purpose
    
- its own identity (Telos)
    
- explicit rules and constraints
    
- approved Patterns
    
- approved Agents
    
- independent continuity
    

Worlds do not share memory or authority by default.

---

### Sessions

A **Session** is a temporary execution envelope.

A Session:

- activates a single World
    
- initializes temporary runtime memory
    
- runs a selected Pattern
    
- produces output
    
- summarizes results
    
- terminates
    

Nothing persists unless explicitly summarized.

---

### Patterns

A **Pattern** is a predefined execution structure.

Patterns:

- define order and pacing
    
- constrain reasoning mode
    
- shape output
    
- do not author memory
    
- do not adapt
    
- do not decide
    

Patterns structure cognition; they do not control it.

---

### Agents

**Agents** are role-bound cognitive executors.

Agents:

- operate only within a World
    
- act only inside a Session
    
- follow the active Pattern
    
- have no autonomous persistence
    
- have no authority
    

Agents reduce Operator load; they do not replace judgment.

---

## Authority and Governance

LOOM enforces a strict separation of roles:

- **Operator** — sole source of intent and decisions
    
- **META** — governance layer (what is allowed)
    
- **A0** — enforcement (applies constraints at runtime)
    
- **ENGINE** — execution plane
    
- **Agents** — bounded cognitive labor
    

No layer may exceed its authority.  
Nothing evolves autonomously.

This separation is non-negotiable.



---

## Memory, Identity, and Continuity

LOOM uses **explicit layered memory**:

- **L4 — Telos**  
    Immutable identity, values, and constraints
    
- **L3 — Knowledge**  
    Reference material and finalized artifacts
    
- **L2 — Episodic**  
    Condensed session continuity
    
- **L1 — Active**  
    Temporary runtime context
    

Higher layers override lower layers.  Nothing persists by accident.

LOOM supports long-horizon projects and optimization **without internalizing objectives or agency**.  Persistence and optimization exist as **Operator-driven, Session-instantiated processes**, not as system-owned drives.

**Identity (Telos) is a ceiling, not a prompt.**  
It cannot be inferred, modified in-session, or drifted into.

## Appendix: Reading LOOM Through an MCP Lens (Non-Canonical)

LOOM was developed independently, but it converges on many of the same concerns as the Model Context Protocol (MCP).

The mapping below is **conceptual**, not structural.

|MCP Concept|LOOM Equivalent|
|---|---|
|Model|ENGINE|
|Context|World|
|Session|Session|
|Instructions|Pattern|
|Tools|Execution Methods|
|Guardrails|META → A0|
|Memory|L2 / L3 (layered)|
|Identity|Telos (L4)|
|Human-in-the-loop|Operator|

Key differences:

- Context cannot expand implicitly
    
- Tools are not model-owned
    
- Identity is immutable
    
- Memory is explicit and governed
    
- No autonomous learning exists
    

MCP is a **translation lens**, not a dependency.
